# 新项目基础代码目录结构框架

> **⚠️ 状态对齐（2026-07-08 补）**：本文为项目启动前的目录结构设计骨架。KJ 项目已按此骨架落地，核心原则（分层单向依赖、热更新边界清晰、Editor 代码零泄露等）均已贯彻到当前代码目录中。实际目录结构以 `CODEMAP.md` 的 File Index 章节和工程内 `Assets/` 实际布局为准——部分细节与骨架有演化（如 `Framework.Package/` 改为 `Framework/` 平铺），但骨架的整体方向仍有效。
>
> 来源：基于 P37 Unity 客户端项目实践提炼
> 目标：保留成熟的目录划分体系，剥离具体游戏业务，形成可跨项目复用的骨架
> 生成日期: 2026-07-01

---

## 一、设计目标

本文档提取 P37 项目经过验证的目录结构设计，输出一套**新项目可以直接套用的骨架结构**。

### 核心原则

1. **分层单向依赖** — 下层不感知上层，编译边界强约束
2. **热更新边界清晰** — 启动壳与业务代码物理分离
3. **Editor 代码零泄露** — 编辑器和运行时代码通过 `.asmdef` 平台限定严格隔离
4. **资源按类型分层** — 代码层和资源层使用统一的分层命名
5. **第三方物理隔离** — 自研代码与第三方库完全分目录
6. **Feature 标准布局** — 每个功能模块统一 MVVM 目录结构

## 二、Assets/ 顶级目录结构（7 目录）

```
Assets/
├── ScriptsC#/           # ⭐ C# 全部源码（按分层划分）
├── GameRes/             # ⭐ 游戏资源（按类型+分层划分）
├── Framework/           # ⭐ 框架代码（自研核心 + 内部包 + 第三方库）
├── Plugins/             # 平台原生插件（.aar/.framework/.jar）
├── StreamingAssets/     # 首包资源（随包体发行的 AssetBundle）
├── Editor/              # 项目级编辑器工具（不归属特定层的独立编辑器脚本）
└── Resources/           # 保留目录（通常由 Addressables 或 AssetBundle 替代）
```

### 划分依据

| 目录 | 文件性质 | Unity 处理方式 | 立项时保留 |
|------|----------|---------------|-----------|
| `ScriptsC#/` | C# 源码 (.cs) | 编译为托管 DLL → IL2CPP | ✅ 必有 |
| `GameRes/` | 美术/音效/数据资源 | 导入为 Asset → 打包 AssetBundle | ✅ 必有 |
| `Framework/` | 框架源码（自研+第三方） | 编译为托管 DLL | ✅ 必有 |
| `Plugins/` | 原生二进制 (.aar/.framework) | 跳过 IL2CPP，直接链接 | ✅ 必有 |
| `StreamingAssets/` | 首包 AB + 配置文件 | 原样复制到 APK/IPA | ✅ 按需 |
| `Editor/` | 顶层编辑器 C# 脚本 | 仅 Editor 编译 | ⚪ 可选 |
| `Resources/` | Unity 动态资源 | Unity 自动打包 | ⚪ 可选（通常不用） |

## 三、ScriptsC#/ — 源码分层骨架

### 3.1 标准层级（5 层）

```
ScriptsC#/
├── Boot/                 # 启动壳层 — 不热更，最先执行
├── Core/                 # 引擎基础设施层 — 零业务逻辑
│   └── Core.Editor/      # Core 的编辑器工具
│   └── Core.Aot/         # HybridCLR AOT 补充（如果用 HybridCLR）
├── General/              # 通用业务层 — 可跨项目复用
│   └── General.Editor/   # General 的编辑器工具
├── Project/              # 项目专属业务层
│   ├── Project.Base/     # 基础业务系统
│   ├── Project.Characters/ # 角色/养成系统（可选，按业务拆）
│   ├── Project.Fight/       # 战斗系统（可选）
│   ├── Project.Gameplay/    # 玩法系统（可选）
│   └── Project.Editor/   # Project 的编辑器工具
├── TableConfig/          # 自动生成的配置表代码
├── API/                  # AI 辅助开发的模块索引文档
└── _AIDocument/          # AI 开发规则/指南文档
```

### 3.2 分层职责定义

| 层级 | 热更 | 职责 | namespace | 立项时必须 |
|------|------|------|-----------|-----------|
| **Boot** | ❌ 不进热更包 | 进程入口、HybridCLR 启动、热更下载、软重启 | `Boot` | ✅ 启动壳 |
| **Core** | ✅ 热更 DLL | 纯引擎系统：Asset/Net/UI/Event/IoC/MVVM/Log/Time/Audio/Storage | `Core` | ✅ 引擎基建 |
| **General** | ✅ 热更 DLL | 通用业务：Login/Role/Item/Mail/RedDot/Server/Chat | `General` | ✅ 通用业务 |
| **Project** | ✅ 热更 DLL | 游戏专属：英雄/装备/抽卡/战斗/副本/商城/公会 | `Project` | ✅ 项目业务 |
| **TableConfig** | ✅ 热更 DLL | Excel→C# 自动生成的配置数据类 | 分包 | ✅ 配置表 |
| **API/** | - | AI 可调用的模块文档索引 | - | ⚪ 可选 |
| **_AIDocument/** | - | AI 开发规则（Rules/Guide） | - | ⚪ 可选 |

### 3.3 依赖流向（强制单向）

```
Boot ──▶ Core ──▶ General ──▶ Project（Base → Characters → Fight / Gameplay）
  │        │         │
  └────────┼─────────┴──▶ TableConfig（被所有层引用，但不引用任何层）
           │
           └──▶ Framework（UniCore/Package/External，被所有层引用）
```

**不可违反的约束**：
- Core 不能引用 General / Project
- General 不能引用 Project
- Boot 只能引用 Framework/UniCore（不能引用 Core/Project，因为热更 DLL 尚未加载）

### 3.4 每层的 Editor 约定

```
{层级}/          → Runtime Assembly（打包进 App）
{层级}.Editor/   → Editor Only Assembly（只在 Unity 编辑器编译）
```

所有 Editor 代码必须放在 `*.Editor/` 目录下，通过 `.asmdef` 的 `Editor` 平台限定确保不出现在正式包中。

### 3.5 Assembly Definition 命名规范

每个子目录对应一个 `.asmdef`，遵循以下命名：

| 目录 | .asmdef 名称 |
|------|-------------|
| `Boot/` | `Boot.Entry` / `Boot.Update` / `Boot.GameLife` |
| `Core/` | `Core` |
| `Core.Editor/` | `Core.Editor` |
| `Core.Aot/` | `Core.Aot` |
| `General/` | `General` |
| `General.Editor/` | `General.Editor` |
| `Project.Base/` | `Project.Base` |
| `Project.Characters/` | `Project.Characters` |
| `Project.Fight/` | `Project.Fight` + 子 Assembly `Project.FightCalculation.*` |
| `Project.Gameplay/` | `Project.Gameplay` |
| `Project.Editor/` | `Project.Editor` |
| `TableConfig/` | `TableConfig` |

### 资源内部分层规则

每种资源类型在其内部 **统一按 Boot/Core/General/Project 分层**：

```
{资源类型}/
├── Boot/               # 启动必需资源
├── Core/               # 引擎/通用 UI 资源
├── General/            # 通用业务资源
└── Project/            # 项目专属（可再拆 Base/Characters/Fight/Gameplay）
```

### 资源加载权限

资源属于哪一层，只能被对应层的代码加载 —— 在配置表中做路径约束：
- Core 代码只能 `Audio/Core/` 下的音频
- General 代码可以加载 `General/` + 继承的 `Core/`
- Project 代码可以加载 `Project/` + `General/` + `Core/`

| `.proto` | Core.Net + 各层 Network 模块 | `build_pb.bat` 生成 C# 代码 |

## 九、Feature 标准目录模板（MVVM）

每个功能模块使用统一的目录结构：

```
{Feature}/
├── Model/              # {Name}Module.cs 或 {Name}Model.cs
├── ViewModel/          # {Name}ViewModel.cs
├── View/               # {Name}Form.cs, {Name}Form.Binding.cs, {Name}Node.cs
├── Data/               # {Name}Data.cs, {Name}Event.cs
├── Util/               # {Name}Util.cs
├── #Generated/         # 自动生成代码（proto stub / config table）— 不手动修改
└── _Test/              # 单元测试
```

### MVVM 层级约束

| 层 | 基类 | 命名 | 限制 |
|----|------|------|------|
| Model (带登录逻辑) | `BaseLogicModule` | `{Name}Module.cs` | 不持有 Unity Object 引用 |
| Model (纯数据) | `BaseModel` | `{Name}Model.cs` | 不持有 Unity Object 引用 |
| ViewModel | `BaseViewModel` | `{Name}ViewModel.cs` | 不持有 Unity Object 引用 |
| Form (全屏/弹窗) | `CSharpForm<TVM>` | `{Name}Form.cs` | 不直接访问 Model |
| Node (列表项/组件) | `CSharpNode<TVM>` | `{Name}Node.cs` | 不直接访问 Model |
| Scene | `CSharpScene<TVM>` | `{Name}Scene.cs` | 不直接访问 Model |

### 数据流

```
Model ──R3 ReactiveProperty──▶ ViewModel ──R3 Binding──▶ View (Form/Node/Scene)
```

### 生命周期

```
Model/ViewModel → DefaultDisposables（软重启时释放）
Form → AutoR3Disposables（OnClose 时释放）
Node → AutoR3Disposables（OnDisable 时释放）
```

## 十一、关键设计决策记录

### 为什么 Boot 层不热更

HybridCLR 加载热更 DLL 的代码本身必须在主包中。Boot 层包含 `GameRestart`（软重启）逻辑，它需要在不退出进程的情况下卸载旧 DLL、加载新 DLL——这个能力本身需要常驻在 AOT 编译的主包中。

### 为什么 Calculation 和 Presentation 分离

战斗系统最关键的设计是将**纯逻辑计算**（`FightCalculation`）与**渲染表现**（`FightPresentation`）分离。Calculation 使用 `FixedPoint` 定点数，确保 Android/iOS/PC 计算结果完全一致。Presentation 只负责播放动画/特效/音效，即使表现层出 Bug 也不影响战斗结果的正确性。

### 为什么 Resources/ 目录为空

Unity `Resources/` 目录中所有资源在打包时都会被无条件包含且无法分包，这会严重增加首包体积。项目使用 `StreamingAssets/` + AssetBundle + 热更新下载的方案替代。

### 为什么 Plugins/ 单独目录

`.aar` / `.framework` / `.jar` 不进 IL2CPP 转换流程，Unity 对 `Plugins/` 目录有特殊处理逻辑。分离后也便于平台原生开发人员独立工作。

### 为什么 GameRes 内部也按层分层

每种资源（Audio/UI/SpineAni/Video 等）内部都按 `Boot/Core/General/Project` 分层，这样：
- 配置表可以精确约束每层代码能加载的资源范围
- 资源依赖和代码依赖方向一致，不会出现 Core 代码依赖 Project 资源的情况
- 分包时可以按层级精确控制首包内容
