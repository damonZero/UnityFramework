# 对象池与缓存系统 — 架构深度评估与重构方案

> 基于 `对象池与缓存系统_代码审查_2026-07-08.md` 的再评估，及全部 22 个源文件的完整阅读。
> 日期：2026-07-08

---

## 一、总体结论

**当前方案不是最优方案。** 它是一份"正确但分裂"的实现——Cache 和 ObjectPool 各自在微观层面写得不错，但两个本该紧密协作的系统在架构层面几乎没有真正整合。

**核心矛盾**：花了一半的工程精力实现了一套可插拔的缓存淘汰策略体系（`ICacheEvictionPolicy` / `Cache<TKey,TValue>` / `LruCachePolicy`），然后只用它做了一个 prefab 资源引用字典（一个任何 `Dictionary` 都能胜任的工作）；而真正需要策略管理的 **GameObject 实例池** 却用裸 `Dictionary + Stack` 实现，完全裸露在策略体系之外。

---

## 二、逐模块深度评估

### 2.1 Cache 模块（`Framework.Cache`）

**好骨架，三处关键缺失。**

#### 问题 2.1.1 — `ICacheEvictionPolicy` 接口粒度过粗

`Touch` 承担了"新 key 加入"和"已有 key 访问"两个完全不同的语义，导致：
- LRU 策略被迫用 if-else 分叉判断是否已存在
- 策略被迫维护内部 `_nodes` 字典做状态判断——这个字典本质上是 `Cache._values` 的索引投影，属于**数据冗余**
- 两个字典的一致性靠 Cache 在锁内按正确顺序调用 `Touch/Remove` 来保证，是**脆弱的隐性契约**
- 无法表达 Peek（只读不更新 recency）语义

**建议改为三方法**：

```cs
public interface ICacheEvictionPolicy<TKey>
{
    void OnAdded(TKey key);       // 新条目加入
    void OnAccessed(TKey key);    // 已有条目被访问
    void OnRemoved(TKey key);     // 条目被移除
    bool TrySelectEvictionCandidate(out TKey key);
    void Clear();
}
```

由 Cache（而非策略）决定调用 `OnAdded` 还是 `OnAccessed`——Cache 本来就知道（它已经查了 `TryGetValue`）。

#### 问题 2.1.2 — `PutUnsafe` 的覆盖语义混乱（对应原审查 B2）

```cs
_values[key] = value;    // 旧值直接丢失，不触发 onEvicted
_policy.Touch(key);      // 策略只知道一次 Touch，不知道值换了
```

三个问题同时发生：
1. **旧值不触发 `onEvicted`**（原审查 B2）——资源泄漏
2. **策略不知道值变了**——对按 value 大小淘汰的策略（如 LFU）是致命语义错误
3. **与 `GetOrAdd` 的 double-check 丢弃路径行为不一致**——一个走 eviction 回调，一个不走

应先检测 key 是否已存在，若存在则移除旧值 + 回调 `onEvicted` + 策略 `OnRemoved`，再插入新值 + 策略 `OnAdded`。

#### 问题 2.1.3 — `GetOrAdd` 缺 single-flight（对应原审查 P2）

不是性能优化，而是**语义正确性**问题：工厂有副作用时（分配 native memory、打开文件句柄），并发 miss 的败者结果即使触发 `onEvicted`，也无法保证副作用可回滚。应 per-key 做 inflight 去重，让同 key 的并发 miss 共享同一个 factory 执行。

#### 问题 2.1.4 — Cache 被架空

唯一调用方 `GameObjectPool` 只用它做 prefab 资源引用的去重和容量控制。这个场景一个 64 容量的 `Dictionary` 就能胜任。Cache 模块存在的价值非常薄。

---

### 2.2 `ObjectPool<T>`（`Framework.Pool.ObjectPool`）

**小而美，但过于孤立。**

- LIFO Stack + `maxIdle` + 锁内外分离 → 实现质量好
- 与 Cache 体系**完全没有任何关系**——没有 `ICacheEvictionPolicy` 参数、不引用任何 Cache 类型
- `maxIdle` 是策略的退化形式（容量策略），但无法与 TTL 等策略**组合**

**建议**：保留 `maxIdle` 作为简单场景的快捷参数，但通过 `ICacheEvictionPolicy` 注入支持策略组合。例如 `maxIdle=200` + `TTL=30s` 同时生效。

---

### 2.3 `GameObjectPool`（`Framework.Pool.Unity.GameObjectPool`）

**整个体系中最需要重构的模块。**

#### 问题 2.3.1 — 五个字典的维护成本

```cs
Dictionary<string, Stack<GameObject>>    _idle;
Dictionary<string, HashSet<GameObject>>  _instancesByPath;
Dictionary<string, int>                  _idleCountByPath;
Dictionary<string, int>                  _activeCountByPath;
HashSet<string>                          _cachedPrefabPaths;
```

五个集合表达的是**同一个概念实体**（prefab 路径下的实例状态）。每次 Get/Recycle/Clear 都要手工同步多个集合，状态一致性完全靠调用顺序保证。`_cachedPrefabPaths` 和 `_prefabCache` 存的是同一份数据。

**应合并为一个内聚结构体** `PrefabPoolState`，只保留一个 `Dictionary<string, PrefabPoolState>`。

#### 问题 2.3.2 — `RemoveInstanceFromRegistry` 的 null 处理（对应原审查 B1）

方法签名的职责分裂——既要从 HashSet 移除一个实例（需要非 null），又要清理空条目（与 null 无关）。当 `instance == null` 时，第一个职责无法执行，但第二个仍然应该做。

应在 `Clear()` 等调用方加 `CleanupEmptyRegistryEntries()` 单独处理，而不是在 `RemoveInstanceFromRegistry` 里 `if (instance == null) return`。

#### 问题 2.3.3 — 线程安全策略不明确（对应原审查 D2/D9）

建议**明确设计为仅主线程**，方法入口加运行时断言，比文档约束更可靠：
```cs
[Conditional("UNITY_EDITOR"), Conditional("DEVELOPMENT_BUILD")]
private static void AssertMainThread() { ... }
```

---

### 2.4 `ResourceCache`（`Framework.Cache.ResContainer`）

**零调用方、无淘汰语义、命名有误导性。** 建议删除。如果未来需要"纯记忆化容器"，放在 Pool 体系下以 `MemoizationStore` 命名。

---

### 2.5 `CollectionPool` + `PooledCollections`

**实现正确**。加 `[NonCopyable]` 即可防御 struct 值拷贝导致的双重归还。

---

### 2.6 测试覆盖

- `PoolTests.cs`：覆盖了 `ObjectPool<T>` 的核心路径（Rent/Return/Preload/MaxIdle/Lease），**缺少 GameObjectPool 测试**
- `CacheThreadSafetyTests.cs`：锁外回调验证高质量，**但被测值类型是 string，不覆盖工厂有副作用或 TValue 持有非托管资源的场景**

## 三、重构方案

### 3.1 目标架构

```
┌──────────────────────────────────────────┐
│  IPool<T>                                │
│  统一表面：Rent / Return / Lease         │
└────────────────┬─────────────────────────┘
                 │
┌────────────────┴─────────────────────────┐
│  ObjectPool<T>                           │
│  LIFO Stack + factory/reset              │
│  构造参数：IBoundedStore<T>（可选）      │
│  装饰器：Stats / Tracking / Lease        │
└────────────────┬─────────────────────────┘
                 │
┌────────────────┴─────────────────────────┐
│  IBoundedStore<TKey, TValue>             │
│  有界对象存放（当前 Cache 的泛化）       │
│  驱动方：IStoreEvictionPolicy<TKey>      │
└────────────────┬─────────────────────────┘
                 │
┌────────────────┴─────────────────────────┐
│  IStoreEvictionPolicy<TKey>              │
│  OnAdded / OnAccessed / OnRemoved        │
│  TrySelectEvictionCandidate             │
│                                          │
│  实现：LruPolicy / FifoPolicy /          │
│         TtlPolicy / CapacityPolicy /     │
│         CompositePolicy                  │
└──────────────────────────────────────────┘
```

### 3.2 关键设计决策

| # | 决策 | 说明 |
|---|------|------|
| 1 | **去除 Cache 作为独立模块** | 当前 Cache 就是"有界字典+淘汰策略"，这是 Pool 的基础设施，不需要独立命名空间 |
| 2 | **策略接口拆为三方法** | 消除策略内部冗余字典，明确 Peek/Replace 等语义 |
| 3 | **CompositePolicy 替代"两者组合"** | `maxIdle` + `TTL` 是两个策略的组合，不是 Pool 的两个独立参数 |
| 4 | **GameObjectPool 五字典合并** | `PrefabPoolState` 内聚所有 per-prefab 状态 |
| 5 | **删掉 ResourceCache** | 零调用方，不保留 |
| 6 | **GameObjectPool 断言主线程** | 运行时检查替代文档约束 |

### 3.3 分步迁移

#### Step 1：修 Bug，不改架构（1–2 天）

- 修 `Cache.Put` 覆盖已有 key 时回调 `onEvicted`（对应 B2）
- 修 `GameObjectPool.Clear()` 中 null 实例注册泄漏（对应 B1）
- `PooledX` struct 加 `[NonCopyable]` 源码注解或 Roslyn analyzer 检测（对应 D5）
- 删除 `ResourceCache` / `ICacheResContainer` / 空目录 / `IPoolable` 死代码
- 给 `GameObjectPool` 补基础测试用例

#### Step 2：重构策略接口 + 引入 IBoundedStore（2–3 天）

- `ICacheEvictionPolicy` → `IStoreEvictionPolicy`，拆为 OnAdded/OnAccessed/OnRemoved
- 新建 `BoundedStore<TKey, TValue>`（当前 Cache 的改进版），修复 §2.1 所有问题
- 加 single-flight 支持
- 旧的 `Cache` 标记 `[Obsolete]`，内部转发到 `BoundedStore`
- 加 `CompositePolicy` + `TtlPolicy` + `CapacityPolicy`
- `GameObjectPool` 的 `_prefabCache` 改用 `BoundedStore`

#### Step 3：GameObjectPool 深度重构（2–3 天）

- 合并五个字典为 `Dictionary<string, PrefabPoolState>`
- 实例 idle store 通过 `IBoundedStore` 接口注入（默认无策略 = 等价旧行为）
- 加 `[MainThread]` 断言
- `SynchronizationPolicy` 显式化（`ObjectPool<T>` 多线程 / `GameObjectPool` 仅主线程）
- 补全 `GameObjectPoolTests`

## 四、与原审查文档的差异点

本评估认同原审查的全部缺陷发现，以下为补充或不同意见：

| 话题 | 原审查 | 本评估 |
|------|--------|--------|
| Cache 与 Pool 的整合 | §8.2-① 建议键控池 vs 单桶池二分 | 进一步：即使单桶池也能从策略组合受益（maxIdle + TTL），不是二分而是统一接口 |
| 策略接口 | 未涉及 | `Touch` 粒度过粗，应拆为 OnAdded/OnAccessed/OnRemoved |
| Put 覆盖语义 | B2 指出缺少 onEvicted | 进一步：不仅是缺回调，策略也不知道值变了，应拆为 Remove + Add 两步 |
| GetOrAdd single-flight | P2 标记为性能优化 | 上升到语义正确性问题（工厂副作用不可回滚） |
| GameObjectPool 重构 | §7.3 三步演进 | 更激进：五字典合并为 PrefabPoolState，建议 Step 3 一次性完成而非两阶段 |
| ResourceCache | 未做决定 | 建议删除 |
| 线程安全 | §8.2-⑥ SynchronizationPolicy | 同意核心方向，补充：GameObjectPool 应做运行时断言而非文档约束 |
