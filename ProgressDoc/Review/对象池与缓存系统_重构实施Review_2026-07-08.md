### 2.3 Unity 对象销毁的 HashSet 陷阱（重点）

> 任务假设：「`state.Instances.Remove(inst)` / `_instanceToPath.Remove(instance)` 在 inst 为已销毁对象时可能找不到条目而移除失败，导致测试 `GetAsyncConsumingDestroyedInstanceDoesNotLeakNullRegistryEntry` 反射断言 `Instances.Count == 1` 在 Editor 中误报失败」。

**独立核验结论（与假设不同）：在 Unity Editor 中该测试不会误报失败，Remove 实际能成功。** 依据：
- Unity 销毁（`Destroy`/`DestroyImmediate`）后，C# 托管包装仍存在，**实例 ID 在托管对象生存期内保持稳定**；`UnityEngine.Object` 重写的 `GetHashCode`/`Equals` 基于实例 ID，故 `HashSet<GameObject>.Remove` / `Dictionary.Remove` 能命中并移除。

**追踪 `GetAsyncConsumingDestroyedInstanceDoesNotLeakNullRegistryEntry`**（`GameObjectPoolTests.cs:58-89`）：
- `DestroyImmediate(a)` 后 `a` 是 fake-null（托管引用非空、实例 ID 稳定）。
- 第二次 `GetAsync` 弹出 `a` → `inst == null` 真 → `UnregisterInstance(a)` → `state.Instances.Remove(a)` **成功**（稳定 ID）→ `Instances` 清空、state 整条移除；随后新建 `b` 重新登记。
- 反射断言 `Instances.Count == 1` 取的是新 `b` 的 state，**等于 1，断言通过**。`AreNotSame(a,b)`、`GetActiveCount==1` 亦通过。

**真正的隐患（非 Remove 失败，而是强引用残留）：**
1. `_instanceToPath` 与 `Instances` 对已销毁实例（尤其是**未在 Recycle 中归还、被外部销毁的 active 实例**）**持有强引用**，阻止其托管包装被 GC——轻微托管内存泄漏，直至 `Clear()`。池行为本身仍正确（不崩溃、不泄漏活对象）。
2. **版本脆性**：测试依赖「Unity 销毁后实例 ID 稳定」这一实现细节；若未来/其他 Unity 构建使 `GetHashCode` 返回 0 或抛异常，`Remove` 将静默失败 → `Instances.Count` 变 2 → 断言失败。即测试对 Unity 内部行为存在脆弱耦合。

**具体修复建议：**
- 测试改用**行为断言**而非内部集合计数：例如「消费已销毁 idle 实例后，下一次 `GetAsync` 返回非 null、与旧实例 `ReferenceEquals` 不同、且 `GetActiveCount==1`」（测试已部分覆盖，移除 `Instances.Count==1` 的反射断言即可）。
- `UnregisterInstance` / `Clear` 在 `Remove` 前用显式假 null 守卫 `if (inst == null)`（注意：fake-null 下 `inst == null` 为真，C# 真 null 亦为真，二者都安全）——当前代码已达此效果，可保留并加注释说明。
- 对确实已销毁且未走正常归还路径的对象，考虑在 `Recycle` 入口或定期巡检中主动 `Remove` 其反向索引条目，避免强引用残留（LOW-2）。

## 5. Blocker / High 具体修复建议

### H1 — Put 后可能被立即淘汰（High）
**根因**：`PutUnsafe` 淘汰循环不排除「当前正在 Put 的 key」，无排序策略会把它当候选。
**修复（最小改动，`BoundedStore.cs`）：**
```csharp
private void PutUnsafe(TKey key, TValue value, ref List<(TKey, TValue)>? evictions)
{
    // ... 覆盖旧值逻辑不变 ...

    _values[key] = value;
    _policy.OnAdded(key);

    // 保护刚 Put 的 key，淘汰循环不得选中它（否则破坏 Put 不变量）。
    while (_capacity > 0 && _values.Count > _capacity)
    {
        if (!_policy.TrySelectEvictionCandidate(out var evictKey) || evictKey.Equals(key))
        {
            // 无可淘汰项，或策略只返回刚 Put 的 key：
            // 说明该策略无法在容量内保留最新写入（如纯 reservoir 且返回自身），
            // 此时强制跳过自身并重试一次；仍无其它候选则打破循环（避免死循环）。
            if (evictKey.Equals(key))
            {
                // 让策略再选一个真正更旧的项；若策略不支持，则接受超限（不再保证容量）。
                if (!_policy.TrySelectEvictionCandidate(out evictKey) || evictKey.Equals(key))
                    break;
            }
            else
                break;
        }

        if (_values.TryGetValue(evictKey, out var evictValue))
        {
            _values.Remove(evictKey);
            _policy.OnRemoved(evictKey);
            evictions ??= new List<(TKey, TValue)>();
            evictions.Add((evictKey, evictValue));
        }
        else
        {
            _policy.OnRemoved(evictKey);
        }
    }
}
```
**更优（治本）**：将 `CapacityPolicy` 改为 FIFO（用 `Queue<TKey>` 记录插入序，`TrySelectEvictionCandidate` 返回队首最旧项），从根本上不返回最新 key；并在 `BoundedStore` 文档/契约中明确「策略不得返回本次 Put 的 key」。两处任选其一即可闭合 H1；建议二者都做。

### H2 — single-flight 工厂异常中毒（High）
**根因**：owner 计算或二次检查路径在 factory 抛异常时跳过 `_inflight.Remove(key)`，faulted `Lazy` 残留。
**修复（`BoundedStore.cs` `GetOrAdd` owner 段，用 try/finally 保证清理）：**
```csharp
var isOwnerLocal = isOwner;            // 捕获是否本线程为 owner
try
{
    var computed = ownerLazy.Value;    // 若 factory 抛异常，跳至 finally
    List<(TKey, TValue)>? evictions = null;
    TValue finalValue;
    lock (_gate)
    {
        if (_values.TryGetValue(key, out var existing))
        {
            finalValue = existing;
            evictions = new List<(TKey, TValue)> { (key, computed) };
        }
        else
        {
            PutUnsafe(key, computed, ref evictions);
            finalValue = computed;
        }
    }
    InvokeEvictions(evictions);
    return finalValue;
}
finally
{
    if (isOwnerLocal)                   // 仅 owner 负责清理 inflight 槽
    {
        lock (_inflightGate)
        {
            _inflight.Remove(key);
        }
    }
}
```
非 owner 路径亦可在 `try/finally` 中包裹并确保不缓存异常态（当前 `ownerLazy.Value` 读 faulted 会复抛，属预期，但 owner 槽已清，后续调用可重新计算）。

### M3（编译确认 —— 已发生并闭合）

用户在实际 Unity 2022.3 工程编译时**确实触发了 M3 担心的场景**：`[UnityEngine.NonCopyable]` 报 CS0246（`NonCopyable`/`NonCopyableAttribute` 均找不到）。

- 根因：本工程编译上下文的 `UnityEngine` 命名空间下无 `NonCopyable`；`manifest.json` 亦未引用 `com.unity.collections`（其 `Unity.Collections.LowLevel.Unsafe.NonCopyable` 才是真编译期阻止拷贝的特性）。
- 已修复：`PooledCollections.cs` 在 `Framework.Pool` 命名空间内定义**本地标记特性 `NonCopyableAttribute`**（`[AttributeUsage(AttributeTargets.Struct)]`），五个 `PooledX` 的 `[NonCopyable]` 解析到该本地特性，编译通过。注释中明确写出升级路径：若需真编译期保护，在 manifest 引入 `com.unity.collections` 并改用 `Unity.Collections.LowLevel.Unsafe.NonCopyable`。
- 局限（如实记录）：本地标记特性**不提供编译期拷贝阻止**，仅作契约声明；D5 的"值拷贝双归还"防护因此退化为"文档/纪律约束"。若要硬防护，需引入 `com.unity.collections`（会增加 Framework 对 Collections 包的依赖，需权衡）。

## 6. 一句话总结

架构重构方向正确、B1/B2/D1/D5 与 C-2/C-3/C-4 均已实质性落地；但独立审查暴露 **H1（Put 不变量在 reservoir 类策略下被破坏）** 与 **H2（single-flight 异常中毒）** 两个 High 级原语正确性缺陷，**H1 须在合并前修复**。第 3 点 HashSet 陷阱经验证在 Unity Editor 中**不会**导致测试误报失败（实例 ID 稳定，Remove 成功），真实风险仅为强引用残留与测试对 Unity 内部行为的版本脆性——建议以行为断言替代内部计数断言。

## 7. 修复附记（2026-07-08 当轮补修）

独立审查发现的 **H1、H2 已在审查当轮修复并回源确认**：

- **H1（store 级防御）**：`BoundedStore.PutUnsafe` 淘汰循环增加 `EqualityComparer<TKey>.Default.Equals(evictKey, key)` 守卫，遇到刚 Put 的 key 直接 `break`，绝不淘汰刚写入的值（破坏「Put 后 key 必存在」不变量前退出）。
- **H1（policy 级治本）**：`CapacityPolicy` 由无序 `HashSet` reservoir 改为 **FIFO**（`LinkedList<TKey>` 记录插入序 + `HashSet` 维护成员），`TrySelectEvictionCandidate` 永远返回最旧 key，从根本上不会选回刚 Put 的新 key，且容量不会临时溢出（消除了报告 §5 H1 建议中"容量为 1 边界可能临时超出 1"的退化）。
- **H2**：`GetOrAdd` owner 路径改用 `try/finally`，无论 factory 是否抛异常都执行 `lock(_inflightGate){ _inflight.Remove(key); }`，杜绝 faulted `Lazy` 残留导致的 key 永久故障、后续同 key 复抛缓存异常。

三者均为 `BoundedStore.cs` / `CapacityPolicy.cs` 局部改动，未触动其它文件，亦未改变对外 API 语义；`GameObjectPool` 仍用 `LruPolicy`（原本安全），但 `BoundedStore` 作为 §10.5「可复用通用原语」现已对 reservoir 类策略同样安全。

**更新判定**：H1、H2 已闭合，且用户在真实工程中触发的 **M3（`[UnityEngine.NonCopyable]` CS0246）已修复**（本地标记特性 `Framework.Pool.NonCopyableAttribute`，编译通过）。**当前已无阻塞项，可进入 Unity Editor 跑 `GameObjectPoolTests` / `BoundedStoreTests` / `CacheThreadSafetyTests` 做最终验证并合并。** 唯一遗留的"非功能"局限：D5 的真编译期拷贝阻止需引入 `com.unity.collections`（见上文 M3）。
