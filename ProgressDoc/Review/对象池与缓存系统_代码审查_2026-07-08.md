# 对象池系统 & 缓存系统 Code Review

> 范围：`Assets/Framework/Pool/**` 与 `Assets/Framework/Cache/**`，以及集成点 `Assets/Scripts/Core/PoolService.cs`
> 日期：2026-07-08
> 结论：**架构清晰、抽象合理，线程安全的基本盘（ObjectPool / Cache 内部锁）是扎实的**；但 `GameObjectPool` 与 `Cache.Put` 各有一个真实缺陷，另有若干健壮性与一致性问题需要收口。

---

## 1. 架构概览

**对象池（Framework.Pool，asmdef 仅引用 `UniTask` + `Cache`）**
- `IPool<T>` / `IPoolLease<T>`：池与租约契约。
- `ObjectPool<T>`：泛型对象池，LIFO（Stack），线程安全（lock + Interlocked）。
- `CollectionPool` + `PooledList/HashSet/Queue/Stack/Dictionary`：集合池（struct 包裹，`IDisposable`）。
- `TypePool`：按 `Type` 注册的全局单例池。
- `PoolLease<T>`：租约式自动回收（Dispose 归还）。
- `Unity/GameObjectPool`：Unity GameObject 专用池，依赖 `Framework.Cache` 做 prefab 引用缓存。
- `PoolDependencies`：静态委托桥，把「资源加载/释放」从 Framework 反向桥到 Core（符合「Framework 不引用 Assets/Scripts」的分层约束）。

**缓存（Framework.Cache，asmdef 无引用，纯基础层）**
- `ICache<TKey,TValue>` / `ICacheEvictionPolicy<TKey>`：缓存与淘汰策略契约。
- `Cache<TKey,TValue>`：带容量 + 淘汰 + `onEvicted` 回调的通用缓存，线程安全。
- `LruCachePolicy<TKey>`：O(1) LRU（LinkedList + Dictionary），所有访问都在 `Cache` 的锁内。
- `ResourceCache<TKey,TValue>` / `ICacheResContainer`：无容量、无淘汰的纯记忆化容器（当前**无任何调用方**）。

**集成点**：`PoolService`（Core 层）注入 `PoolDependencies.LoadAssetAsync/ReleaseAssetByPath`，持有唯一 `GameObjectPool(null, 64)`。

> 注：本节描述**当前实现**。关于「缓存 = 容器 + 策略 / 池 = 在策略容器中取还对象并叠加附加功能」的**理想模型、现状偏差与优化建议**，见 §7、§8。

---

## 2. 优点（值得保留）

1. **正确的异步双重检查加载**：`GameObjectPool.GetAsync` 用 per-path `SemaphoreSlim(1,1)` + 二次 `TryGet` 防重复加载 prefab，逻辑正确。
2. **回调在锁外执行**：`Cache` 的 `onEvicted` 与 `GetOrAdd` 的 factory **都设计在锁外运行**，并通过 `CacheThreadSafetyTests` 四个用例验证（死锁/重入场景）。这是高质量设计，测试也写得到位。
3. **LRU 实现正确且 O(1)**：`Touch` 复用节点、`TrySelectEvictionCandidate` 取尾节点；`Cache.PutUnsafe` 的淘汰循环对「策略返回不在字典中的 key」做了兜底 `Remove` + 重试，不会死循环。
4. **`ObjectPool<T>` 统计与回收分离**：`_reset` 在 `lock` 外调用，降低锁竞争。
5. **分层合规**：`Pool.asmdef` 不引用任何 `Assets/Scripts`；`PoolDependencies` 用静态委托反向桥接，正确规避了 Framework→Project 的依赖方向限制。

## 3. 缺陷清单（按严重度）

### 🔴 中危（建议尽快修）

| # | 位置 | 问题 |
|---|------|------|
| **B1** | `GameObjectPool.GetAsync`（约 L38-62） | **空实例泄漏 + Clear 时 Destroy(null)**。当从 `_idle` 弹出的实例已被 Unity 销毁（`inst == null`，假 null）时，代码调用 `RemoveInstanceFromRegistry(prefabPath, inst)` 但该方法首行 `if (instance == null) return;` 直接 no-op，导致该实例的条目永久残留在 `_instancesByPath` 中。后续 `Clear()` 遍历 `_instancesByPath` 会对该 null 调用 `Object.Destroy(inst)`（Unity 报 warn/error）。应改为在 null 分支真正移除注册项（或在 `RemoveInstanceFromRegistry` 内允许按 key 清尾）。 |
| **B2** | `Cache.Put` → `PutUnsafe`（L160-182） | **覆盖已有 key 时旧值不走 `onEvicted`**。`_values[key] = value;` 直接覆盖，旧值被静默丢弃而**不触发回收回调**。对于任何「值需要释放/反注册」的缓存（如持有非托管/Unity 资源的缓存），这是资源泄漏。应：若 key 已存在，把旧值加入 `evictions` 一并回调。对比 `Remove`/`GetOrAdd` 淘汰路径都会回调，`Put` 覆盖路径不一致。 |
| **D1** | `GameObjectPool.Recycle`（L99-132） | **idle 实例无上限、永不修剪**。`Recycle` 总是 `Push`，没有 per-prefab `maxIdle`，也没有全局实例数上限或定时回收。会话中不同 prefab 各自保留若干 idle 实例，只有 prefab *引用* 受 `Cache` 的 64 上限约束，**GameObject 实例本身会无限增长**，长会话存在内存膨胀风险。建议加 per-prefab `maxIdle` 与/或总量上限 + 修剪。 |
| **D5** | `PooledList/HashSet/Queue/Stack/Dictionary<T>`（PooledCollections.cs） | **可变 struct + 值拷贝导致双重归还**。这些 `PooledX` 是 mutable struct，`Dispose` 仅置 `_value=null` 防自身二次 dispose。但若被**值拷贝**（`var b = a;`）后两个副本都 `Dispose`，第一个置 null、第二个仍持有原 `_value`，会把同一集合**两次归还进共享池**→ 池损坏（同一 list 存两份）。早期 .NET Pooled 类型同款坑。建议加 `[UnityEngine.NonCopyable]` 让拷贝在编译期报错，或在注释/文档中明确「禁止拷贝，必须 `using`/ref 使用」。 |

### 🟡 低~中危（健壮性问题）

| # | 位置 | 问题 |
|---|------|------|
| **D2** | `GameObjectPool` 全局 | **单线程假设未设防**。`_idle / _instancesByPath / _idleCountByPath / _activeCountByPath` 均为裸 `Dictionary/Stack/HashSet`，在 `async` 方法中无锁读写。Unity + UniTask 默认在主线回复，故"当前能跑"，但 `await gate.WaitAsync(...)` 与 Unity 对象生命周期（加载中 Destroy）使该假设很脆：若在 await 期间发生 `Clear`（如 Shutdown 与 Warmup 并发），状态会损坏。建议要么明确文档约束"仅限主线程"，要么对可变状态加锁/改并发集合。 |
| **D9** | `GameObjectPool.OnPrefabEvicted`（L265-272） | 该回调是 `Cache` 的 `onEvicted`，运行在**缓存锁外**，却会修改 `GameObjectPool._cachedPrefabPaths`（裸 HashSet）。与 D2 同源：多线程下存在竞态。 |
| **D10** | `GameObjectPool.Warmup`（L134-142）+ `PoolService.Shutdown` | `Warmup` 用 `.Forget()` 火忘式预热。若 `Shutdown`（`PoolDependencies.LoadAssetAsync = null` + `Clear`）在预热 await 期间发生，`LoadPrefabAsync` 会在被遗忘的 task 中抛 `InvalidOperationException` → **未观测异常**。建议预热携带 CancellationToken 或在 Shutdown 时先等预热完成/取消。 |
| **D3** | `GameObjectPool.GetAsync`（L65-97） | prefab 加载返回 `null`（资源缺失）时仅返回 null、**无日志**。调用方必须判空且难以排查，建议补一条 warning 日志。 |
| **D4** | `IPoolable`（Pool/Interfaces/IPoolable.cs） | **死代码**。`ResetState()` 接口定义后从未被引用；`ObjectPool` 用的是 `Action<T>?` 委托。要么在 `ObjectPool` 中自动识别 `IPoolable`（`reset==null && item is IPoolable p` → `p.ResetState()`），要么删除该接口，避免误导。 |
| **D7** | `TypePool.Register` / `GetOrCreate`（TypePool.cs L10-40） | `Register` 静默覆盖；`GetOrCreate` 虽双检，但并发首调可双双建池，败者池泄漏（其 preload 被静默丢弃）。属初始化期问题，主线程下概率低，可加个 double-checked 收口。 |
| **D6** | `Pool/Objects/` 与 `Pool/Collections/` | 空脚手架目录（仅 .gitkeep），属遗留。建议删除或补充内容。 |

### 🔵 信息项

- `ObjectPool` 与 `Cache` 的 `capacity=0` / `maxIdle=0` 均表示「无上限」，但内部初始容量取 4。语义一致，但建议在 XML 注释里明确写明「0 = 无限」，避免误用。

## 6. 性能专项评估 & 设计合理性（补充，19:06）

> 聚焦 `Framework/Cache` 的运行时开销与设计取舍。结论先行：**当前生产用法（仅 `GameObjectPool` 的 prefab 引用缓存，容量 64，低频）下没有实际性能问题；但作为"可复用通用缓存"，设计偏保守，存在可预见的并发读瓶颈与两处设计缺口。**

### 6.1 命中路径的真实开销（热路径分析）
一次 `TryGet` 命中路径（`Cache.cs:41-54`）：
1. `lock(_gate)` 进入独占锁；
2. `Dictionary.TryGetValue`（O(1) 平均）；
3. `_policy.Touch(key)` —— LRU 必须**在每次读取时也修改策略状态**：`LruCachePolicy.Touch`（`LruCachePolicy.cs:11-22`）做 `_order.Remove(node)` + `_order.AddFirst(node)` 两次 LinkedList 改写 + 一次 Dictionary 查找；
4. `unlock`。

即**纯读取也是一次写入**（改 LRU 链表）。这是 LRU 算法的固有约束，不是 bug，但它决定了：**读路径无法无锁化，也无法用 ReaderWriterLockSlim 提速**（"读锁"下照样要改链表）。

### 6.2 性能结论
- **无 O(n) 扫描、无命中路径堆分配、LRU 为 O(1)** —— 没有"性能 bug"。
- **唯一实质瓶颈 = 单把全局独占锁（`_gate`）**。读多写少的高并发场景会在此串行化（lock convoy，争用时可能从数十 ns 退化到 μs 级）。
- **当前用法不受影响**：`GameObjectPool` 仅在"新 prefab 路径首次加载"时触碰 `_prefabCache`，且 prefab 加载本身是异步 I/O（远比锁慢），并发不同路径的 `TryGet/Put` 争用相对于 I/O 可忽略。**所以今天没有性能问题。**
- `GetOrAdd` 采用"锁外跑 factory + 二次double-check"（`Cache.cs:63-99`），factory 不持锁（已测），但**未做 single-flight**：并发 miss 同 key 时两个线程都会跑 factory，败者结果被丢弃（`onEvicted` 回调，已在测试中验证）。对昂贵 factory 是浪费。

### 6.3 设计合理性
**合理之处（保留）**
- 策略模式干净：`ICache` 与 `ICacheEvictionPolicy` 分离，`Cache` 不依赖具体淘汰算法，可无痛扩展 FIFO / LFU / TinyLFU。
- 回调与 factory **都在锁外执行**（`Put`/`GetOrAdd` 先收集 `evictions` 列表，出锁后 `InvokeEvictions`），避免重入死锁 —— 设计正确且被 `CacheThreadSafetyTests` 4 用例背书。
- LRU 实现正确且 O(1)（节点复用，非每 key 重建）。

**设计缺口（按优先级）**
1. **P2 — 单锁模型不适合"读多写少"复用场景**。若未来要做高频/并发缓存，正确升级路径是**近似 LRU（CLOCK / Clock-Pro）**：用 `Interlocked` 时间戳记录访问、后台或惰性淘汰，而非简单换成 `ReaderWriterLockSlim`（对真 LRU 无效）。当前设计作为低频缓存 OK。
2. **P2 — `GetOrAdd` 缺 single-flight**。同 key 并发 miss 重复执行 factory。建议加 per-key 计算锁或 `Task/ValueTask` 记忆化，避免重复昂贵创建 + 丢弃浪费。
3. **P3 — 仅支持"条目数"容量，无 cost/size 预算**。`capacity` 是条目计数，无法按"对象大小/内存占用"淘汰。缓存大体量对象（贴图/资源引用）时粒度太粗。建议 `ICacheEvictionPolicy` 或 `Cache` 增加可选 `long cost` 维度。
4. **P3 — `ResourceCache` / `ICacheResContainer` 设计存疑**。它是**无容量、无淘汰**的纯记忆化容器（`ResourceCache.cs`），当前**零调用方**。作为"Cache"它没有任何缓存语义（不淘汰、只增不减 → 内存只涨），命名与方法（`GetOrCreate`/`TryRemove`/`Clear`）易让人误用。建议二选一：**(a) 删除**（属未落地的脚手架），或 **(b) 让其组合 `ICacheEvictionPolicy`** 真正成为"带淘汰的资源容器"，否则保留即误导。
5. **P3 — 无"只读不更新 recency"的 Peek API**。某些场景（探测性查询）不应影响 LRU 顺序，`ICache` 缺 `TryPeek`。

### 6.4 一句话判定
> 对**它现在做的事**（低频 prefab 引用缓存）设计完全合理、性能无忧；作为**框架级通用缓存**它是"正确但保守"的实现，主要天花板是单锁 + 真 LRU 的读改写的固有耦合，外加 `ResourceCache` 这处半成品。若要面向高频/并发复用，再按 6.3 的 P2 项演进即可，不必现在动。

## 7. 设计意图 vs 当前实现偏差（用户原意，19:55）

> 本节记录用户对这两套系统的**设计意图（ideal）**，并对照当前实现标明偏差。结论：当前实现与用户的原始设想在「**策略可组合性**」与「**池 / 缓存职责边界**」两处存在明显偏差——当前代码只是该设想的一个**局部雏形**。

### 7.1 设计意图（理想模型）

1. **缓存系统 = 容器 + 策略（两层）**
   - **容器层** `Cache<TKey,TValue>`：一个有界、可被策略驱动的「对象存放容器」，按 spec（prefab 路径 / 类型 / key）持有**可复用对象**。
   - **策略层** `ICacheEvictionPolicy<TKey>`：决定「容量满时淘汰哪个对象」，可插拔（LRU / FIFO / LFU / TTL / maxIdle…）。
   - 缓存系统本身**不关心对象是什么**，只负责「按策略存 / 取 / 淘汰」。它是纯基础原语，依赖自由。

2. **对象池 = 在「含缓存策略的容器」中 取 / 还 复用对象**
   - 对象池**包裹（wrap）一个带策略的缓存容器**。
   - `Get`：从容器取出一个可复用对象（miss 时构造 / 加载），交给调用方。
   - `Return` / `Recycle`：把用完的对象**返还进容器**，供下次复用。
   - 即：**复用发生在「池」这一层**，容器只提供「有界 + 策略淘汰」的存放能力。

3. **所有对象池在构建时灵活指定策略**
   - 策略是**构造参数**，不是写死的。
   - 无论是 `ObjectPool<T>`、`CollectionPool`、`TypePool` 还是 `GameObjectPool`，都应在构造时传入想用的 `ICacheEvictionPolicy`，从而灵活选择 LRU / FIFO / TTL 等。
   - 这要求「策略可组合」是**全部池**的共性能力，而非某个池的私有细节。

4. **对象池扩展附加功能（池层价值）**
   - 对象池在「取 / 还」之上，提供缓存容器本身不具备的、与「对象生命周期」相关的附加能力：
     - **对象标记追踪（marking / tracking）**：在途 / 空闲 / 激活状态、实例登记表、生命周期标记。
     - **对象初始化重置（init / reset）**：取用时 `OnGet` 重置 / 初始化，归还时 `OnRecycle` 清理，保证复用对象处于干净状态。
     - 其他：预热（warmup / preload）、统计、租约（lease）、校验等。
   - 这些附加功能属于「池」的领域职责，**不应塞进缓存容器**。

### 7.2 当前实现偏差清单

| 偏差 | 理想 | 当前实现 | 关联 |
|---|---|---|---|
| **G1 策略未向所有池开放** | 所有池构造时可指定策略 | 仅 `GameObjectPool` 一个类碰了 Cache，且 `new LruCachePolicy` **写死**；`ObjectPool` / `CollectionPool` / `TypePool` 完全不碰 Cache | 「组合不同策略」只是设计意图，未落地 |
| **G2 容器持有的是 prefab 引用，不是可复用对象** | 容器持有「可复用对象」，池从容器中取 / 还对象 | `GameObjectPool` 的 `Cache<string,GameObject>` 缓存的是 **prefab 资源引用**；可被复用的 GameObject **实例**在 `_idle`（`Dictionary<Stack>`）里，**无策略、无上限** | 真正该被策略管理的「实例」反而在策略体系之外（即 D1） |
| **G3 池未真正 wrap 缓存容器** | 池 = 带策略容器的包裹者 | 池与缓存是「顺手用一下」的关系（仅 prefab 字典），实例复用逻辑与缓存体系脱钩 | 职责边界模糊，缓存的「有界 + 策略」能力没用在刀刃上 |
| **G4 附加功能不统一** | 标记追踪 / 初始化重置 是所有池的标配扩展 | `IPoolable.ResetState()` 是**死代码**（D4）；只有 `GameObjectPool` 有实例登记表（`_instancesByPath`）；`ObjectPool` 靠 `Action<T>` 回调做 reset；`CollectionPool` / `TypePool` 无对象生命周期语义 | 附加能力零散、无统一契约，复用对象状态清理不可靠 |

> **一句话**：用户设想的「缓存 = 容器 + 策略；池 = 在容器中取还对象并叠加附加能力；所有池可灵活选策略」是**清晰且自洽的目标架构**；当前实现**只在 `GameObjectPool` 的 prefab 引用这一小角落**部分触碰了该模型，且策略写死、实例侧无策略、附加功能无统一契约。

### 7.3 目标架构草图（演进方向 / 北极星）

```
[ 缓存系统（原语，无领域知识） ]
   Cache<TKey,TValue>            ← 有界容器，按 spec 持有「可复用对象」
   ICacheEvictionPolicy<TKey>    ← 可插拔策略：LRU / FIFO / LFU / TTL / maxIdle
        ▲ 构造时注入
[ 对象池（组合者 + 附加层） ]
   Pool<T> ──wrap──> Cache<TKey,TValue>(policy)
     Get    : 从容器取对象 → 触发 初始化重置 (OnGet)
     Return : 对象 → 标记追踪 → 返还容器 (触发 回收重置 / OnRecycle)
     + 预热 / 统计 / 租约 / 校验
```

**演进步骤（建议顺序）**
1. **开放策略注入**：给 `ObjectPool` / `CollectionPool` / `TypePool` / `GameObjectPool` 增加 `ICacheEvictionPolicy<TKey>` 构造参数（默认 LRU 以保兼容），`GameObjectPool` 去掉硬编码 `new LruCachePolicy`。
2. **让容器持有可复用对象**：将 `GameObjectPool._idle` 的「per-prefab 实例 Stack」改造成由带策略的 `Cache` / 容器驱动（即把 D1 的 `maxIdle` 泛化为「实例层的淘汰策略」），使「取 / 还」真正发生在策略容器中。
3. **统一附加功能契约**：定义 `IPoolObject`（或接管现有 `IPoolable`）的生命周期接口（`OnGet` / `OnRecycle` / `MarkState`），所有池在 Get / Return 时统一调用；删除或接管当前死代码的 `IPoolable`；把 `GameObjectPool` 的实例标记追踪提炼为池层通用能力。
4. **补齐策略实现**：新增 `FifoCachePolicy` / `TtlCachePolicy` / `MaxIdleCachePolicy`，让「所有池灵活选策略」成为现实。

### 7.4 偏差结论
> 当前实现是用户设想的**一个局部雏形**：策略模式接口已就位，但只被一个池在极小范围使用且写死 LRU；真正的「容器持有可复用对象 + 所有池组合策略 + 池层统一附加功能」尚未建立。建议把 §7.3 作为后续重构的**北极星**，其优先级高于 §4 的修 Bug 项（B / D 类可边改边靠齐该目标）。

## 8. 设计评估与优化建议（20:03）

> 基于 §7 的设计意图与偏差，本节对「缓存 = 容器 + 策略 / 池 = 在策略容器中取还对象并叠加附加功能」这一设想做评估，并给出可落地的优化建议。**结论：方向正确、可落地，是比现状好得多的目标形态**；下面只挑能把它从「正确」推到「精致」的点。本节是对 §7 的**细化与优化**，二者构成「理想 → 偏差 → 优化」闭环，不矛盾（§8.2-① 是对 §7 策略注入原则的抽象细化，见 §8.5）。

### 8.1 评估：值得肯定的部分（保留）
1. **容器/策略 与 池/生命周期 的切分清晰**，符合开闭原则，且能复用现有 `ICacheEvictionPolicy` 这个 seam。
2. **构造期注入策略**，使所有池可测、可演进。
3. **「池负责取/还、容器只管有界+淘汰」的职责划分**，天然避免让缓存容器去理解「对象是什么」。

### 8.2 优化建议（按性价比）
1. **策略分两类，勿强塞 keyed 策略给所有池**（对 §7「所有池构造时指定策略」的细化，不矛盾：原则一致，抽象分形）
   - **键控池**（`GameObjectPool` 按 prefab 路径、`TypePool` 按 Type）→ 用 `ICacheEvictionPolicy<TKey>`（淘汰哪个 key）。
   - **单桶池**（`ObjectPool<T>`、`CollectionPool`，每类型一个池）→ 其「策略」实为**容量/修剪策略**（保留几个 idle），应是对 `Cache`/后备存储的简单 reservoir + `maxIdle`，而非 keyed eviction。
   - 统一在 `IPool` 表面下，但内部不强行同构。
2. **附加功能用「装饰器」而非「池内方法」**
   - 核心 `CorePool` 只管 取/还 + 工厂；tracking / stats / lease 做成**可选装饰器**包在外面，按需开关、易单测，避免池变成胖类。
3. **生命周期契约 = 接口 + 委托 + `IsValid`（顺带修 B1）**
   - 定义 `ILifecyclePolicy<T>`（`OnGet` / `OnRecycle` / `OnDestroy` / `IsValid`），**自动探测**：对象实现接口就调，未实现走 `Action<T>` 委托兜底（保留现有灵活性，接管当前死代码的 `IPoolable`）。
   - `IsValid` 让取用时发现失效对象（如 Unity 对象被销毁）即重建 → 把 **B1（null 实例泄漏）收进生命周期契约**，不再是零散的 if。
4. **`GameObjectPool` 应组合「两个存储」**
   - **实例库**（按 prefab 路径，容量/TTL 修剪 idle 实例）= 你设想的「容器持有可复用对象」。
   - prefab **资源引用**本身也该被缓存（避免重复加载资源）→ 另一个键控 LRU 缓存。
   - 理想 `GameObjectPool` = 资源引用缓存(LRU) + 实例库(容量/TTL) 的组合。现状只有前者、后者（D1）缺失。
5. **默认用「租约」归还，降低泄漏**
   - 已有 `PoolLease<T>` / `IPoolLease<T>`；把租约作为**默认归还机制**（`using`），手动 `Return` 作 opt-out，机制上杜绝「取了忘还」。
6. **显式「同步策略」，决定锁归谁**
   - `ObjectPool<T>` 可能多线程（已有 lock+Interlocked），`GameObjectPool` 仅主线程（锁多余）。若池 wrap 带锁容器，锁会重叠/语义不清。
   - 把同步模型也作为池的显式维度（`SynchronizationPolicy: None / Lock / Concurrent`），容器保持简单，由池决定是否加锁。

### 8.3 YAGNI 提醒
别为「所有池都能选策略」建一个无人配置的 `IPoolStrategy` 超大继承树。只有关键控池真正需要键控淘汰策略；单桶池给 `maxIdle` 数字即够。先让关键路径（`GameObjectPool` 的实例库 + prefab 缓存）吃上策略，其余保持简单——比一步到位的大重构更稳。

### 8.4 目标形态（文字版）
```
IPool<T>                    // 统一表面：Get / Return / Lease
  └─ CorePool<TKey,T>       // 最小核心：取/还 + 工厂
       ├─ IObjectStore      // 后备存储：键控 Cache | 单桶 reservoir
       │    └─ 策略：键控淘汰(LRU/FIFO/TTL) | 容量(maxIdle)
       └─ ILifecyclePolicy<T>   // OnGet/OnRecycle/OnDestroy/IsValid（接口+委托）
  可选装饰器（按需包）：TrackingPool<T>  StatsPool<T>  LeasePool<T>
  GameObjectPool = 资源引用缓存(LRU) + 实例库(容量/TTL) 的组合
```

### 8.5 与本文档其他章节的关系
- §7 给出「理想模型 + 现状偏差」；本节在其基础上给出**细化与优化**，二者构成「理想 → 偏差 → 优化」闭环，非矛盾（§8.2-① 是对 §7 策略注入原则的抽象细化，§7.3 步骤 1 的「开放策略注入」在 §8 中被拆分为「键控淘汰 / 容量」两类）。
- 优化项可与缺陷对应落地：**③ `IsValid` → 修 B1**；**① 容量策略 → 修 D1**；**② 装饰器化 tracking → 补 D2/D9 的追踪一致性**；**⑤ 租约 → 缓解取还泄漏类隐患**。建议与 §4 的修 Bug 项并行、边改边靠齐 §7.3 北极星。

### 10.2 重构方案比本文档更尖锐的新发现（建议补入本文档）

重构方案在本文档基础上，给出了几个**更深一层**的论断，应并入后续分析：

1. **`Touch` 粒度过粗（§2.1.1，比本文档新）**：`LruCachePolicy.Touch`（`LruCachePolicy.cs:11-22`）用 if-else 同时表达「新 key 加入」「已有 key 访问」两个语义，被迫维护内部 `_nodes` 字典（本质是 `Cache._values` 的索引投影 → **数据冗余**），两个字典一致性靠 Cache 锁内按序调用 `Touch/Remove` 的**隐性契约**保证。建议拆为 `OnAdded / OnAccessed / OnRemoved / TrySelectEvictionCandidate / Clear` 三方法接口（→ 见 §10.5 的 `IStoreEvictionPolicy`）。
2. **`Put` 覆盖不仅是「缺回调」（§2.1.2，比 B2 更深）**：B2 只指出旧值不触发 `onEvicted`；重构方案指出更深的语义错误——**策略也不知道值变了**，对按 value 大小淘汰的策略（LFU）是致命错误。正确做法是拆为「`Remove` 旧值 → `onEvicted` → 策略 `OnRemoved` → `Add` 新值 → 策略 `OnAdded`」两步。
3. **single-flight 不是性能而是正确性（§2.1.3，比 §6.2 P2 更准确）**：工厂有副作用（分配 native memory、开文件句柄）时，并发 miss 败者结果即使 `onEvicted` 也无法回滚副作用 → 应 per-key 做 inflight 去重。本文档 §6.2 将其标为「性能优化」，此定性**偏低**，建议升级为语义正确性。
4. **`GameObjectPool` 五字典合并（§2.3.1）**：`_idle / _instancesByPath / _idleCountByPath / _activeCountByPath / _cachedPrefabPaths`（`GameObjectPool.cs:16-20`）表达的是**同一概念实体**（per-prefab 实例状态），每次操作靠调用顺序手工同步。建议合并为单一 `Dictionary<string, PrefabPoolState>`，显著降低状态不一致风险（也收紧了本文档 D2/D9 的追踪一致性问题）。
5. **主线程运行时断言（§2.3.3）**：`GameObjectPool` 单线程假设应以 `[Conditional]` 运行时断言强制，而非仅靠文档约束（强化本文档 §8.2-⑥ 的 `SynchronizationPolicy`）。

### 10.3 关键分歧：缓存系统是否应作为「独立模块」

这是本文档（`§7` 用户意图）与重构方案**唯一正面冲突**之处，必须显式调和：

| | 立场 | 依据 |
|---|---|---|
| **重构方案 §3.2-#1** | **去除 `Cache` 作为独立模块**，吸收进 Pool 作为 `BoundedStore`，「它只是 Pool 的基础设施，不需要独立命名空间」 | 当前唯一调用方 `GameObjectPool` 只用它做 prefab 引用字典，一个 `Dictionary` 即可胜任，Cache 模块价值薄（§2.1.4） |
| **§7 用户设计意图** | **`Cache = 容器 + 策略` 应作为可复用的独立原语**，所有对象池在构造时组合不同策略 | 用户明确：「缓存系统就是容器以及策略，对象池负责在包含缓存策略的容器中取出和返还复用对象」 |

**研判（推荐）**：二者并非二选一，可按如下方式调和——

- 重构方案的「Cache 价值薄」批评，基准是**当前用法**（只被 `GameObjectPool` 的 prefab 字典用到）。而 §7 用户意图恰恰是要**扩展** Cache 的用途（所有池组合、实例库存也由策略容器驱动）。若采纳 §7，`Cache` 会从「薄」变「厚」，独立模块的合理性随之成立。
- 重构方案的**技术重塑**（`BoundedStore` + `IStoreEvictionPolicy` 三方法）与 §7 的「容器 + 策略」模型**结构一致**；争议的只是这个抽象放在 `Framework.Cache` 还是 `Framework.Pool` 命名空间下。
- **建议保留 `Cache` 为独立模块**，但按重构方案重塑其接口与实现：
  - 命名进化为 `BoundedStore<TKey,TValue>` + `IStoreEvictionPolicy<TKey>`（三方法）；
  - 修复 §10.2 的所有语义问题（Put 两步、Touch 拆分、single-flight）；
  - 补 `TtlPolicy` / `CapacityPolicy` / `CompositePolicy`；
  - 它作为**通用有界 KV 原语**，不仅对 Pool 有用（资源句柄缓存、记忆化等场景亦可复用），保留独立模块收益更大。
- 即：**采纳重构方案的全部技术修复 + 抽象重塑，但在模块归属上保留 `Cache` 独立模块（采纳 §7 用户意图）**。这同时满足「Cache 干净好用」与「所有池灵活组合策略」两个目标。

### 10.4 设计谱系洞察：KJ 当前实现是「祖先设计的退化版」

结合 `ProgressDoc/Discuss/对象池/` 三篇历史分析（`37项目-...深度分析`、`ETPro-...深度分析`、`对象池方案对比总结`），可解释重构方案所说的「花一半精力做策略体系，却只用在 prefab 字典」这一核心矛盾：

- **37 项目（KJ 池设计的祖先）**：`UnityObjectPool<T> = MultiAssetContainer<T> + IStrategy<string>`，**策略本来就是挂在「实例容器」上的**——`Recycle` 时 `strategy.AfterPut(uid) → 容量检查 → OnEviction → container.EvictionByStrategy(uid)` 直接淘汰 idle **实例**。`Cache` 包与策略层是配套存在的。
- **KJ 当前 `GameObjectPool` 退化了**：继承了「prefab 引用用 `Cache`+LRU 缓存」这一半，却**丢了「实例容器也套策略」**那一半——实例直接进裸 `_idle` Stack（无策略、无上限，即 D1 / G2）。
- 因此重构方案的「核心矛盾」准确说应是：**KJ 从祖先那里继承了一套完整的 策略+容器 机制，但只把策略接到了 prefab 缓存，没接到实例库存**。这**恰好印证 §7 G2 与 §8.2-④**——让「实例库存本身 cache-backed」不是新发明，而是**把祖先丢掉的设计 reclaim 回来**。
- **ETPro 的借鉴点**（对比总结 §二）也佐证 §7：`goPool`（prefab 资产）+ `instCache`（实例 LIFO）**双层分离**、`instPathCache` 反向索引污染检测、`persistentPathCache` 常驻保护、`checkCanPopFunc` 淘汰前「活跃实例不淘汰」——这些都应作为 `GameObjectPool` 重构（§10.5）的具体采纳点。

### 10.5 修订后的目标架构（综合重构方案 §3.1 + §7 用户意图 + §10.4 谱系）

```
[ 缓存系统：独立可复用原语（采纳 §7，按重构方案重塑） ]
   BoundedStore<TKey,TValue>           ← 原 Cache，修复 Put/Touch/single-flight
   IStoreEvictionPolicy<TKey>          ← 三方法：OnAdded/OnAccessed/OnRemoved
        实现：LruPolicy / FifoPolicy / TtlPolicy / CapacityPolicy / CompositePolicy

[ 对象池：组合者 + 附加层（§7 / §8） ]
   IPool<T>                             ← 统一表面：Get / Return / Lease
     └─ ObjectPool<TKey,T>             ← 持 IBoundedStore<TKey,T>（构造注入）
          ├─ 策略：键控池(LRU/FIFO/TTL) | 单桶池(Capacity/Ttl via Composite)
          └─ ILifecyclePolicy<T>       ← OnGet/OnRecycle/OnDestroy/IsValid（接口+委托，接管 IPoolable）
   可选装饰器：TrackingPool<T> / StatsPool<T> / LeasePool<T>
   GameObjectPool = prefab BoundedStore(LRU)           ← 已存在，去硬编码
                  + instance BoundedStore(Capacity/TTL) ← reclaim 祖先设计（修 D1/G2）
                  + 反向索引污染检测（借鉴 ETPro instPathCache）
                  + [MainThread] 运行时断言
```

**与 §8.2-① 的调和**：单桶池不再被「排除在策略接口之外」，而是用 `CapacityPolicy`/`TtlPolicy`（经 `CompositePolicy` 组合 `maxIdle + TTL`）统一接入 `IStoreEvictionPolicy`，消除「键控 vs 单桶」的二元割裂——这与重构方案 §四对本文档 §8.2-① 的细化一致。

### 10.6 结论

1. 本文档的缺陷事实（B1/B2/D5、single-flight、ResourceCache 半成品）被重构方案**独立佐证且回源成立**。
2. 重构方案提供了 5 项更深的发现（§10.2），应并入后续分析与改造清单。
3. **唯一冲突点（Cache 是否独立模块）已调和**：采纳重构方案的全部技术修复与抽象重塑，**保留 `Cache` 为独立模块**（符合 §7 用户意图，且 Cache 作为通用有界原语具独立复用价值）。
4. 谱系洞察表明：§7 的「实例库存 cache-backed」不是新设计，而是 **reclaim 37 项目祖先 `UnityObjectPool = 容器 + 策略` 的设计**——KJ 当前实现退化地只接了一半。这进一步支持「保留 Cache 模块 + 把它真正接到实例库存」的方向。

> **最终判定**：两份文档**互补而非冲突**。本文档负责「现状缺陷 + 用户设计意图」，重构方案负责「更深的实现级修复 + 统一目标架构」。二者在「保留 Cache 为独立可复用模块、所有池组合策略、实例库存也由策略驱动」上完全收敛。下一步改造应以 §10.5 为目标架构、以重构方案 Step1–3 为迁移路径。
