# KJ 项目目录结构规范

> 日期: 2026-07-01
> 作用: 规定 Assets/ 下所有目录的划分依据和各类代码/资源的归属位置

---

## 一、命名空间规范

### 核心规则：命名空间 = 目录路径，不带任何项目名前缀

| ❌ 禁止 | ✅ 正确 | 说明 |
|--------|--------|------|
| `KJ.Core.Asset` | `Core.Asset` | 命名空间和项目名无关 |
| `KJ.Boot` | `Boot` | asmdef 的 `rootNamespace` 就是层名 |
| `KJ.General.Models` | `General.Models` | 目录路径即命名空间 |
| `KJ.Framework.Pool` | `Framework.Pool` | Framework 是顶级目录 |

### 原因

1. 代码已经是项目的一部分，不需要在命名空间里再声明"我是 KJ"。
2. `namespace` 唯一的作用是区分符号来源——目录分层 + `.asmdef` 已经解决了这件事。
3. 将来 `Framework/Pool/` 提取为独立 Package 时，命名空间就是 `Framework.Pool`，不能带项目前缀。

### asmdef 命名

| asmdef 文件 | rootNamespace |
|-------------|---------------|
| `Boot.asmdef` | `Boot` |
| `Core.asmdef` | `Core` |
| `General.asmdef` | `General` |
| `Project.asmdef` | `Project` |
| `Framework.Pool.asmdef` | `Framework.Pool` |

现有 asmdef 文件名（如 `KJ.Boot.asmdef`）在以后合适的时间点重命名为不带前缀的版本。

## 一、Assets/ 顶级目录（7 目录）

```
Assets/
├── Scripts/            # ⭐ C# 源码（按分层划分 .asmdef）
├── Framework/          # ⭐ KJ 自有框架包（不含第三方，可独立提取为 UPM Package）
├── GameRes/            # ⭐ 游戏资源（美术/音效/配置/场景，按分层划分）
├── Plugins/            # 平台原生插件（.aar / .framework / .jar）
├── StreamingAssets/    # 首包资源（随 APK/IPA 发行的 AssetBundle）
├── Editor/             # 项目级编辑器工具（不归属特定层的独立编辑器脚本）
├── Resources/          # 保留目录，仅放启动必需的配置（如 AssetConfig.asset）
```

### 划分依据

| 目录 | 文件性质 | Unity 处理方式 | 立项必建 |
|------|----------|---------------|---------|
| `Scripts/` | C# 源码 (.cs) | 编译为托管 DLL → IL2CPP | ✅ |
| `Framework/` | 独立包源码 (.cs) | 编译为托管 DLL（独立 .asmdef） | ✅ |
| `GameRes/` | 美术/音效/数据资源 | 导入 → 打包 AssetBundle | ✅ |
| `Plugins/` | 原生二进制 | 跳过 IL2CPP，直接链接 | ✅ |
| `StreamingAssets/` | 首包 AB + 配置文件 | 原样复制到 APK/IPA | 按需 |
| `Editor/` | 跨层项目级编辑器 C# 脚本 | 仅 Editor 编译 | 按需 |
| `Resources/` | 最小引导配置 | Unity 自动打包 | ✅（仅放 AssetConfig.asset） |

### Editor 代码归属

优先把编辑器代码放在所属模块旁边，而不是全部塞进顶层 `Assets/Editor/`：

```text
Assets/Scripts/Boot.Editor/          # Boot 相关编辑器工具
Assets/Scripts/Core.Editor/          # Core 相关编辑器工具
Assets/Scripts/General.Editor/       # General 相关编辑器工具
Assets/Scripts/Project.Editor/       # Project 相关编辑器工具
Assets/Framework/Asset.Editor/       # Framework.Asset 相关编辑器工具
Assets/Framework/Pool.Editor/        # Framework.Pool 相关编辑器工具
```

规则：
- `*.Editor.asmdef` 必须设置 `includePlatforms: ["Editor"]`。
- Unity 只天然按名为 `Editor` 的文件夹排除 Player 编译；`Boot.Editor` 这类目录必须靠 asmdef 限定 Editor 平台。
- 顶层 `Assets/Editor/` 只放确实跨层、无法归属到某个模块的项目级工具。
- Editor 程序集可以引用 Runtime 程序集；Runtime 程序集禁止引用 Editor 程序集。

### `_` 前缀路径

- 任意路径段以 `_` 开头的文件或文件夹视为私有/临时内容，不参与 Player 包和 YooAsset 资源包。
- Unity 不会天然排除 `_` 前缀路径；资源包必须通过 YooAsset ignore rule 或构建校验执行该规则。
- 当前 `Framework.Asset.Editor.KJAssetIgnoreRule` 会在 YooAsset 收集阶段忽略 `_` 前缀路径段。
- 当前 `Boot.Editor.Build.PlayerBuildPrivatePathValidator` 会在 Player Build 前拦截 Build Settings 场景、`Resources`、`StreamingAssets` 中的 `_` 前缀路径段。

### 为什么不叫 ScriptsC#

项目规模决定了 `Scripts/` 足够。P37 用了 `ScriptsC#/` 是因为同时和 `ScriptsLua/` 共存。KJ 不做 Lua，用简洁命名即可。

## 二、Scripts/ — 源码分层（4 层）

```
Scripts/
├── Boot/          → KJ.Boot.asmdef         (引用: Framework.Asset)
├── Boot.Editor/   → Boot.Editor.asmdef     (Editor-only)
├── Core/          → Core.asmdef            (引用: Framework.Asset/Event + VContainer + MessagePipe + UniTask)
├── General/       → General.asmdef         (引用: Core + Framework.Event)
└── Project/       → Project.asmdef         (引用: Core + General + Framework.Event)
```

### 分层职责

| 层级 | 热更 | 职责 | 命名空间 | 代码约定 |
|------|------|------|----------|----------|
| **Boot** | 当前作为最小启动路径；目标拆分后 `Boot.Update` ✅ 但需重启生效 | 进程入口、最小更新 UI、资源版本检查、HybridCLR 引导 | `Boot` | Entry MonoBehaviour + 序列化启动配置；不创建正式业务容器 |
| **Core** | ✅ | 引擎基础设施：DI、事件、System 管理器、Asset 封装、对象池胶水 | `Core` | `[CoreSystem]` + `ISystem` |
| **General** | ✅ | 通用业务：配置表、音频、红点、本地化 | `General` | `[Model]` + `IModel` |
| **Project** | ✅ | 项目专属：具体玩法、UI 窗口、业务流程 | `Project` | `[Model]` + `IModel` |

Boot 热更边界按更细粒度理解：
- 当前工具默认只把 `Core` / `General` / `Project` 作为正式运行时预加载热更 DLL；`Assets/Scripts/Boot/` 还没有拆出独立的启动更新清单。
- `Boot`/`Framework` 的 C# 改动不是天然必须换包；如果已发布版本具备下载和加载对应托管 DLL 的能力，通常按“热更资源下发 + 重启 APP/下次启动生效”处理。
- 真正必须换包的是 native/player/HybridCLR 底层加载机制、IL2CPP 产物、Java/OC/C++、原生插件或旧包缺少加载能力的情况。
- 目标形态是 `BootLoader/Entry` 保持极薄，`Boot.Update` 承载启动更新流程并作为启动阶段热更 DLL。
- `Boot.Update` 更新后通常需要重启 APP 或下次启动生效；非商业 HybridCLR 基线不支持替换已加载程序集。

### 依赖方向（强制单向）

```
Boot ──▶ Core ──▶ General ──▶ Project
```

- Boot 不引用 Core / General / Project；只允许引用稳定的 `Assets/Framework/` 基础接口和必要启动异步库
- Core 不引用 General / Project
- General 不引用 Project
- 违反 → .asmdef 编译时报错

## 三、Framework/ — KJ 自有框架包

### 定位

存放**业务无关、可独立编译、可单独提取为 UPM Package** 的 KJ 自有框架代码。

> ⚠️ 这是自有代码仓库，不是第三方库目录。第三方库全部通过 UPM（`Packages/manifest.json`）管理。

### 与 Scripts/Core/ 的差异

| | Scripts/Core/ | Framework/ |
|---|---|---|
| **可以引用项目代码？** | ✅ 可以引用 Boot + 其他 Core 系统 | ❌ 不可以 |
| **可以引用 Unity API？** | ✅ | ✅（仅必要的子包） |
| **可以独立提取？** | ❌ 耦合到 Core.asmdef | ✅ 搬走目录 + 它的 .asmdef 即可 |
| **典型内容** | 胶水层：AssetSystem、SystemManager、PoolService、VContainer/MessagePipe 注册 | 稳定接口和底层适配：IAssetSystem、AssetRuntime、GameEventAttribute、ObjectPool、IStrategy、LruCache |
| **举例** | 把 AssetRuntime/ObjectPool 注入到 VContainer 的注册代码 | YooAsset 适配实现、ObjectPool<T> 本身（一个 Stack + lock 的纯 C# 类） |

### 目录结构

```
Framework/
├── Pool/                         ← 对象池（纯逻辑 + Unity 扩展）
│   ├── Collections/              ← 集合租约（List / HashSet / Queue / Stack / Dictionary）
│   ├── Objects/                  ← 纯 C# 对象池
│   ├── Types/                    ← 类型池 / 注册表
│   └── Unity/                    ← Unity 对象池 / 适配层
├── Cache/                        ← 缓存策略 + 资源容器
    ├── Strategy/                 ← 淘汰策略（LRU / FIFO）
    └── ResContainer/             ← 资源容器（GameObject / Component）
├── Asset/                        ← 资源系统统一 API + YooAsset 等第三方资源库适配
├── Event/                        ← 统一事件标记 + 类型扫描（不直接绑定 MessagePipe）
└── TestKit/                      ← 通用测试基础设施、Fake、Probe、Fixture（不放具体测试用例）
```

### Package 不引用项目代码如何工作？

通过**接口、适配器或静态委托注入**。Framework 本身只定义稳定功能逻辑，依赖外部能力时声明接口或委托字段，由 Core 层在启动时注入：

```csharp
// ===== Framework/Pool =====
public static class PoolDependencies
{
    // GameObject 池创建实例时需要加载 Prefab
    public static Func<string, Transform, UniTask<GameObject>> LoadAssetAsync;
    public static Action<Object, Transform> ReleaseAsset;
}

// ===== Scripts/Core/ =====
public class PoolService : ISystem
{
    public void Init()
    {
        // 注入 Core 的实现
        PoolDependencies.LoadAssetAsync = (path, parent) => 
            _assetSystem.LoadAssetAsync<GameObject>(path);
    }
}
```

### 将来抽成 UPM Package 时的操作

1. 把 `Framework/Pool/` 目录搬到一个独立 Git 仓库
2. 在新的仓库根目录放一个 `package.json`
3. 项目中通过 `manifest.json` 的 git URL 引用
4. Core 层的 `PoolService.cs` 不需要搬——它属于胶水层，留在 Core

## 四、GameRes/ — 游戏资源目录

### 标准结构

```
GameRes/
├── Audio/               # 音频（BGM/SFX/语音）
│   ├── Core/             # 引擎公共音效
│   ├── General/          # 通用 UI 音效
│   └── Project/          # 玩法专属音效
├── UI/                   # UI Prefab + 图集
│   ├── Boot/             # 启动 UI
│   ├── Core/             # 引擎公共 UI（Loading/对话框/Toast）
│   ├── General/          # 通用业务 UI（登录/邮件/设置）
│   ├── Project/          # 项目专属 UI
│   ├── ResConfig/        # UI 资源配置（图集配置、字体映射等 ScriptableObject）
│   ├── ResLocalization/  # 多语言 UI 资源
│   └── ResPool/          # UI 对象池配置
├── Character/            # 角色模型（Hero/NPC/Monster）
├── Scene/                # Unity 场景
│   └── Boot/             # 启动场景（最小场景，极快加载）
├── Effects/              # 视觉特效
├── Shaders/              # 自定义 Shader
├── Font/                 # 字体（SDF/位图）
├── SpineAni/             # Spine 2D 动画
├── FrameAni/             # 序列帧动画
├── Video/                # CG 视频
├── Config/               # ScriptableObject 运行时配置
├── Common/               # 通用共享资源
├── Protobuf3/            # .proto 协议定义
│   ├── core/
│   ├── general/
│   └── project/
└── _Temp/                # 临时工作区（打包前清理）
```

### 资源内部按层分层

每种资源类型内部统一按 Core / General / Project 分层，加载权限对应：

| 资源路径 | 代码加载权限 |
|----------|-------------|
| `{类型}/Core/` | 仅 Core 代码 |
| `{类型}/General/` | General + Core 代码 |
| `{类型}/Project/` | 所有层代码 |

## 五、各类资产归属速查表

| 我要放... | 放哪里 | 理由 |
|-----------|--------|------|
| 纯 C# 对象池 `ObjectPool<T>` | `Framework/Pool/Objects/` | 零依赖，纯逻辑 |
| 集合租约 `PooledList/HashSet/Queue/Stack/Dictionary` | `Framework/Pool/Collections/` | `using` 归还，必须保留 |
| GameObject 池（加载 Prefab 实例化） | `Framework/Pool/Unity/` | Pool 包内，通过委托桥接加载 |
| LRU / FIFO 淘汰策略 | `Framework/Cache/Strategy/` | 策略层，可独立测试 |
| 资源系统统一接口与第三方适配 | `Framework/Asset/` | 上层不直接依赖 YooAsset 等第三方库 |
| 事件标记与事件总线统一入口 | `Framework/Event/` | 上层不直接绑定 MessagePipe 细节 |
| 通用测试 Fake / Probe / Fixture / NUnit 扩展 | `Framework/TestKit/` | 可复用测试基础设施，不放具体测试用例 |
| 池系统的 DI 注册 + 全局入口 | `Scripts/Core/Pool/PoolService.cs` | 胶水层，引用 VContainer + Framework.Asset |
| AI 运行日志稳定接口/配置 | `Framework/Log/` | 只定义 `GameLog`、`IGameLogSink`、配置、早期缓冲等稳定契约，不引用 Core/ZLogger/UnityEngine |
| AI 运行日志文件管线 | `Scripts/Core/Logging/` | 生成 session、写 JSONL、接入 ZLogger/ILogger、flush、维护 latest 指针 |
| AI 日志分析/导出 Editor 工具 | `Scripts/Core.Editor/Logging/` 或 `Framework/Log.Editor/` | Core 负责运行日志收集；Framework.Log.Editor 负责模块树/配置面板 |
| HybridCLR 启动加载 | `Scripts/Boot/` | Boot 侧 AOT 引导：Entry 序列化配置、资源更新、AOT metadata 与热更 DLL 加载 |
| HybridCLR 构建同步工具 | `Assets/Scripts/Boot.Editor/HybridCLR/` | Editor-only：调用 HybridCLR/YooAsset Editor API，生成并同步热更 RawFile 资源 |
| YooAsset 项目忽略规则 | `Assets/Framework/Asset.Editor/YooAsset/` | Editor-only：资源系统打包规则，例如 `_` 前缀路径不参与收集 |
| HybridCLR 热更 DLL / AOT metadata RawFile | `Assets/GameRes/HotUpdate/` | YooAsset 收集并更新；Boot 只通过 Entry 序列化路径读取 |
| Core 启动阶段和 DI 注册 | `Scripts/Core/Bootstrap/` | Core 容器注册扩展、类型扫描注册入口、CoreStartupContext |
| Core 系统生命周期协议/管理器 | `Scripts/Core/Systems/` | 放 `ISystem`、`SystemManager`、`CoreSystemAttribute`、Core 生命周期事件 |
| Project 启动阶段和注册入口 | `Scripts/Project/Bootstrap/` | 正式游戏组合根，创建 VContainer root 并串联 Core/General/Project 注册 |
| UI 窗口 Prefab | `GameRes/UI/General/LoginForm.prefab` | 业务 UI 资源 |
| 引擎公共 UI（Loading遮罩） | `GameRes/UI/Core/LoadingMask.prefab` | 引擎级 UI |
| 启动更新界面 | `GameRes/UI/Boot/` 或 Boot 场景内对象 | 只显示更新进度、重试/修复，不承载登录业务 |
| Luban 配置表代码 | `Scripts/General/Config/` | 通用业务层 |
| `AssetConfig` ScriptableObject | `Assets/Resources/AssetConfig.asset` | 启动时 Resources.Load |
| 第三方 DLL（Protobuf 等） | `Assets/Plugins/` | Unity 特殊处理 |
| 构建管线 Editor 脚本 | 所属模块的 `*.Editor/`；跨层才放 `Assets/Editor/` | 仅 Editor 编译 |
| 编辑器扩展（Inspector/Window） | 所属模块的 `*.Editor/` 或 `Editor/` 子目录 | Editor Only |
| Feature 业务模块（如 Shop） | `Scripts/Project/Shop/` | 项目专属，按需拆 .asmdef |

## 六、代码和 Framework 的引用关系

```
                         被依赖方 (可以引用谁)
              ┌──────┬──────┬──────┬──────┬──────────────┐
              │ Boot │ Core │ Gen  │ Proj │ Framework    │
  ────────────┼──────┼──────┼──────┼──────┼──────────────┤
  依赖  Boot   │  ●   │  ❌  │  ❌  │  ❌  │  ✅ 稳定接口 │
  方    Core   │  ✅  │  ●   │  ❌  │  ❌  │     ✅       │
       General │  ✅  │  ✅  │  ●   │  ❌  │  ✅ 稳定接口 │
       Project │  ✅  │  ✅  │  ✅  │  ●   │  ✅ 稳定接口 │
       Framework│ ❌  │  ❌  │  ❌  │  ❌  │     ●        │
  ────────────┴──────┴──────┴──────┴──────┴──────────────┘

✅ = 允许   ❌ = 禁止   ● = 同层内部
```

关键约束：
- **Framework/** 直接承载稳定底层模块，不使用 `Framework/Package/` 子目录
- **Framework/** 不能引用任何 `Scripts/` 下的汇编
- **Core** 负责桥接：引用 Framework 包，注入委托依赖，注册到 VContainer / MessagePipe
- **General / Project** 可以引用稳定 Framework 接口；业务编排仍优先通过 Core/General 门面完成

## 七、Feature 模块标准目录（MVVM）

每个业务 Feature 使用统一结构：

```
{Feature}/
├── Model/              # 数据模型（不持有 Unity Object 引用）
├── ViewModel/          # 视图模型（不持有 Unity Object 引用）
├── View/               # Form (全屏/弹窗) / Node (列表项)
├── Data/               # 事件定义、数据结构
├── Util/               # 工具函数
├── #Generated/         # 自动生成代码（proto stub / config table）— 不手动修改
└── _Test/              # 单元测试，不参与 Player/YooAsset 打包
```

## 八、立项操作清单

1. ✅ 建 `Assets/Scripts/` 下 4 层 asmdef + 启动骨架
2. ✅ 建 `Assets/Framework/Pool/` + `Cache/` + `Asset/` + `Event/` + `TestKit/` 基础目录
3. ⬜ 建 `Assets/GameRes/` 资源目录骨架（后续按需逐步创建）
4. ✅ 建模块化 Editor 目录（如 `Assets/Scripts/Boot.Editor/`、`Assets/Framework/Asset.Editor/`）；`Assets/Editor/` 仅保留跨层工具占位
5. ⬜ 建 `Assets/Plugins/`（导入原生插件时）
6. ⬜ 建 `Assets/StreamingAssets/`（出首包时）
